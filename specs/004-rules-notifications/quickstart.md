# Quickstart: Rules & Local Notifications

Validation guide for Feature F4. Assumes F1 (auth + seeded categories) and
F2 (expense CRUD + recurrence) are already in place.

## Prerequisites

- Node.js LTS and the Expo CLI (`npx expo` — no global install needed)
- A Supabase project with F1/F2's migrations plus
  `contracts/rules-schema.sql` applied
- A signed-in test account
- A physical device or simulator that supports local notification
  permission prompts (the iOS Simulator and most Android emulators support
  scheduled local notifications)

## Setup

```bash
npm install
npx expo start
```

Sign in with a test account.

## Scenario 1 — Default rules are seeded on first login (User Story 1/2, FR-001)

1. Sign in with a brand-new test account for the first time.
2. Open the Rules tab.
3. **Expect**: exactly two rules exist — "Big expense ahead" (≥ €200, 5 days
   before + 1 day before) and "Due tomorrow" (any amount, 1 day before,
   grouped) — both enabled, both marked as defaults (not deletable).

## Scenario 2 — Big expense reminder is scheduled (User Story 1, FR-006/FR-009)

1. Add an expense: amount €250, due date 10 days from today.
2. **Expect**: a local notification is scheduled for 5 days before the due
   date (check via the device's scheduled notifications, or inspect
   `expo-notifications`' `getAllScheduledNotificationsAsync()` output in a
   debug build).
3. Wait until (or fast-forward device time to) 1 day before the due date
   without marking it paid.
4. **Expect**: a second reminder fires.

## Scenario 3 — Marking paid cancels pending reminders (User Story 1, FR-010)

1. Using the expense from Scenario 2 (before either reminder has fired),
   mark it paid.
2. **Expect**: both the 5-day and 1-day scheduled notifications for that
   expense are cancelled and never fire.

## Scenario 4 — Editing an expense below threshold cancels its reminder (User Story 1, FR-010)

1. Add an expense: amount €250, due in 10 days (as Scenario 2).
2. Edit its amount down to €100.
3. **Expect**: the "Big expense ahead" reminder tied to it is cancelled (no
   longer meets the ≥ €200 threshold). If it's still due tomorrow-or-sooner
   at any point, the "Due tomorrow" rule may still apply independently.

## Scenario 5 — Grouped "due tomorrow" digest (User Story 2, FR-001/FR-011)

1. Add two expenses both due tomorrow: one €50, one €300.
2. **Expect**: exactly one grouped notification is scheduled for today
   mentioning both, not two separate notifications, and the €300 expense's
   own "Due tomorrow"-time trigger from the big-expense rule does not also
   fire separately (Edge Cases, FR-011).

## Scenario 6 — Editing a rule reconciles existing notifications (User Story 3, FR-004/FR-007)

1. With the expense from Scenario 2 style (€250 due in 10 days) still
   pending, open the Rules tab and raise "Big expense ahead"'s threshold to
   €500.
2. **Expect**: the previously-scheduled reminder for that €250 expense is
   cancelled without the user touching the expense itself.

## Scenario 7 — Disabling a rule cancels its notifications (User Story 3, FR-007)

1. With at least one pending reminder from the "Due tomorrow" rule, disable
   that rule from the Rules tab.
2. **Expect**: all notifications generated by that rule are cancelled
   immediately; re-enabling it later causes it to be re-evaluated on the
   next reconciliation (next mutation or app foreground).

## Scenario 8 — Default rules can't be deleted (User Story 3, FR-005)

1. Attempt to delete "Big expense ahead" or "Due tomorrow" from the Rules
   tab.
2. **Expect**: deletion is blocked; the app offers disabling instead.

## Scenario 9 — Custom rule create/delete (User Story 3, FR-003/FR-005)

1. Create a new custom rule: category filter = "Subscriptions" only, any
   amount, 3 days before.
2. Add a Subscriptions expense due in 3 days and a Groceries expense also
   due in 3 days.
3. **Expect**: only the Subscriptions expense gets a reminder from this new
   rule.
4. Delete the custom rule.
5. **Expect**: its reminder is cancelled.

## Scenario 10 — Recurring roll-forward reschedules reminders (User Story 1, FR-015)

1. Create a monthly recurring expense, amount €250, due in 10 days.
2. Mark it paid (rolls forward per F2 to next month, per `mark_expense_paid`).
3. **Expect**: any reminders tied to the original occurrence are gone, and
   fresh reminders are scheduled against the new occurrence's due date.

## Scenario 11 — Notification permission denied (Edge Cases, FR-014)

1. Deny notification permission when prompted (or revoke it in device
   settings, then reopen the app).
2. **Expect**: the Rules screen shows a banner indicating reminders are off;
   expenses and rules remain fully usable (add/edit/delete/mark-paid all
   still work); no crash or blocking dialog.

## Scenario 12 — Ownership isolation (FR-013)

1. Using two different signed-in accounts, confirm neither can see or
   modify the other's rules or notification history (e.g., a direct API
   call with account A's session against account B's rule id returns no
   rows / is rejected by RLS).

## Automated coverage

- `tests/unit/notificationEngine.test.ts` — covers rule matching (amount
  threshold, category filter, enabled/disabled), the past-trigger-point
  skip (FR-009), cross-rule dedupe/grouping (FR-011, Scenario 5), and
  recurring roll-forward rescheduling (Scenario 10), all against plain
  in-memory rule/expense fixtures with no OS mocking.
- `tests/unit/money.test.ts` — covers the extracted `toCents` shared
  helper (if not already fully covered by existing dashboard tests).
- `tests/component/rules-screen.test.tsx` — covers Scenario 1's default
  seeding display, Scenario 6-7's edit/disable-reconciles UI flow,
  Scenario 8's default-rule delete-blocked UI, Scenario 9's custom rule
  create/delete UI, and Scenario 11's permission-off banner.
